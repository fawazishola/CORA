# CORA V2 Integration Plan

This document defines the next version of CORA after the first physical, bootable prototype exists.

`V2` is the first more integrated, more refined, and more execution-serious CORA revision after V1 has reached first boot.

If `V1` proves CORA can exist physically and boot, `V2` proves CORA can mature as a hardware program.

## What V2 Is

`CORA V2` is:

- the first serious integrated revision after lessons from V1
- mechanically cleaner
- thermally smarter
- more coherent as an appliance
- closer to a public-demo and research-grade hardware story

## What V2 Is Not

`CORA V2` is still not:

- mass-production hardware
- the final industrial design forever
- the end of the board-iteration cycle

But it should be the first version that feels like a real hardware program instead of only an unusually strong prototype.

## V2 Objective

Take the lessons from V1 and turn CORA into a more refined appliance with stronger integration across:

- board path
- enclosure
- thermal handling
- serviceability
- boot reliability
- public demonstration readiness

## V2 Strategy

V2 should improve what V1 proves, not restart the project from scratch.

That means:

1. preserve the core design language
2. fix the weak points discovered in V1
3. move closer to the intended carrier-board architecture
4. tighten the mechanical and system integration
5. make the booting-appliance behavior more stable and presentation-ready

## V2 Design Basis

V2 begins only after V1 answers the first real questions:

- does the enclosure geometry work physically?
- do the standoffs and panel logic feel right?
- do the connector positions work?
- does the object look like CORA in real life?
- where are the thermal and assembly pain points?
- what failed or held during first boot?
- how stable is the display and input bring-up path?

Those answers become the design input for V2.

## V2 Target Window

Assuming:

- `V1` is physically assembled and reaches first boot between `April 17, 2026` and `April 24, 2026`

then the realistic target window for `V2` is:

- `Friday, May 15, 2026` to `Friday, June 12, 2026`

That range is realistic for a stronger integrated revision without pretending hardware iteration is instantaneous.

## What Changes From V1 To V2

### 1. Better Mechanical Precision

V2 should improve:

- panel fit
- connector alignment
- fastener consistency
- board suspension and mounting confidence
- cable behavior in the assembled object

### 2. Better Thermal Reality

V2 should explicitly address:

- SoC / FPGA heat concentration
- airflow assumptions
- passive heatsink placement
- whether hidden venting is needed
- whether silent active cooling is required

### 3. Better Boot Reliability

V2 should improve:

- repeatable boot behavior
- cleaner display output path
- more reliable keyboard / input handling
- cleaner debugability during bring-up
- stronger confidence that the appliance behaves like a real machine, not only a successful one-off boot

### 4. Better Internal Integration

V2 should move closer to the intended CORA architecture by tightening:

- board-level layout assumptions
- I/O cleanliness
- debug accessibility
- boot and bring-up path coherence

### 5. Better Public Demonstration Readiness

V2 should be much stronger for:

- photography
- recorded demos
- applications and grants
- partner or faculty conversations
- school features and public presentation

## Two Possible V2 Paths

### V2A: Refined Dev-Board Appliance

This is the faster V2 path.

It keeps the development-board foundation but improves:

- enclosure fit
- thermal logic
- structural precision
- internal presentation

This path is useful if the custom carrier board is not yet mature enough.

### V2B: Early Carrier-Board Integration

This is the stronger but slower V2 path.

It begins moving from the development-board path toward:

- a more custom carrier-board implementation
- more faithful board-level architecture
- better ownership of the physical compute substrate

This path is ideal if the board work matures quickly enough after V1.

## V2 Recommended Path

The most realistic approach is:

- use `V2A` if speed and visual/public readiness matter most immediately
- move into `V2B` as the board architecture becomes ready

That avoids stalling the whole project while waiting for perfect board maturity.

## V2 Timeline

### Late April 2026

- complete V1 lessons review
- document mechanical and thermal problems
- decide whether V2 is `V2A` or `V2B`

### Early May 2026

- revise CAD
- revise panel cutouts
- revise standoff specification if needed
- tighten the integration plan

### Mid May 2026

- order revised mechanical parts
- prepare improved internal layout or board path
- finalize the V2 assembly package

### Late May 2026

- assemble the refined V2 unit
- run better visual, fit, and thermal tests
- produce stronger demo artifacts

### Early June 2026

- reach first real public-demo-ready V2 state

## V2 Procurement Direction

The same sourcing direction remains valid for V2:

- acrylic: `SendCutSend` or `Xometry`
- stainless structural hardware: `McMaster-Carr`
- future board fabrication: `JLCPCB` or `PCBWay`

What changes in V2 is not the vendor philosophy. It is the maturity of the geometry and part choices.

## V2 Deliverables

By the end of V2, CORA should ideally have:

- a cleaner assembled unit
- improved enclosure fidelity
- stronger internal hardware legibility
- better thermal confidence
- a more coherent bring-up story
- public-demo and feature-ready imagery

## V2 Success Criteria

V2 is successful if:

- CORA no longer feels like only a first prototype
- the object is mechanically believable
- the thermal story is no longer mostly speculative
- the system is easier to present as a real hardware program
- the next funding and collaboration conversations become easier

## Practical Read

`V1` is the first undeniable object.

`V2` is the first version that begins to feel like the hardware program is truly underway.
